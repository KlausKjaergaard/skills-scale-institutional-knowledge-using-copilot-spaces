# OctoAcme — Risk Management & Communication

## Purpose
Explain how to identify, manage, and communicate risks and dependencies.

## Risk Register
Maintain a simple table with:
- ID
- Description
- Impact (High/Med/Low)
- Likelihood (High/Med/Low)
- Owner
- Mitigation plan
- Status

## Risk Lifecycle
- Identify: during planning and ongoing execution
- Assess: estimate impact and likelihood
- Mitigate: reduced via actions, contingency plans
- Monitor: review at weekly syncs and update status

## Stakeholder Communication
- **Key Stakeholder Groups** are formally identified at project initiation with documented communication needs (e.g., engineering, sales, support, executive sponsors)
- Assign a primary contact within each stakeholder group to streamline approvals and feedback
- Provide regular updates (weekly or milestone-based) via the **Project Manager**
- Engage **Key Stakeholder Groups** at sprint reviews for direct feedback on delivered features
- Use a single source of truth (project README or release doc) for status
- **Release Manager** notifies stakeholder groups ahead of each planned deployment

## Stakeholder Engagement Model
- **Requirements Collection**: Product Manager and Project Manager lead sessions with Key Stakeholders at project start
- **Progress Reviews**: Stakeholders invited to sprint reviews and milestone demos (facilitated by Scrum Master)
- **Approval Workflows**: Stakeholders formally approve major deliverables and release sign-off through defined channels
- **Escalation**: Stakeholders escalate business-level risks to the Project Manager or Product Manager; unresolved issues are escalated to the sponsor

## Escalation Paths
- Team-level -> Scrum Master -> PM -> Product Lead -> Sponsor
- **Key Stakeholder Groups** escalate business risks directly to PM or Product Lead
- For security incidents, follow the security incident runbook and notify Security on-call
